# Chapter 3: System Analysis

## 3.1 Introduction

System analysis is the phase in which the broad aims of a software project are translated into a precise, verifiable description of what the system must do and the constraints under which it must operate. Before any line of application code is written, it is essential to establish a shared understanding of the problem, the people the system serves, and the qualities the finished product must exhibit. This chapter performs that work for MindMate, the AI-powered mental health assistant for students described in the preceding chapters. It elaborates the requirements gathered from the project brief and the wider literature, classifies them into functional, non-functional and user requirements, examines the feasibility of building the system within the available resources, and models its behaviour through use-case and data-flow analysis. The chapter closes with a statement of the hardware and software needed to develop and run the application.

The analysis follows the structured approach to requirements engineering advocated in the software engineering literature, in which requirements are first elicited, then specified in a form that is unambiguous and traceable, and finally validated against the goals of the project (Sommerville, 2016; Pressman & Maxim, 2014). MindMate is a postgraduate dissertation project rather than a commercial product, so the analysis is scoped accordingly: the emphasis is on demonstrating a complete, internally consistent and reproducible system rather than on satisfying the full demands of a regulated clinical deployment. Throughout, the project's three objectives act as the anchor for the requirements, to design and develop an AI mental-health assistant for students, to personalise its support using machine-learning algorithms, and to evaluate its effectiveness on mental-health and academic outcomes. Every requirement recorded below can be traced back to one of these objectives.

A second concern that shapes the analysis is safety. Because the application addresses psychological wellbeing, the system must behave responsibly when a user expresses distress. The need for a conservative, rule-based crisis-detection layer that runs before any conversational reply is therefore treated not as an optional feature but as a first-class requirement. This reflects the cautionary findings of reviews of mental-health chatbots, which note that conversational agents can offer genuine support but must be designed with explicit safeguards and clear boundaries on what they claim to do (Vaidyam et al., 2019; Abd-Alrazaq et al., 2019).

## 3.2 Requirement Analysis

Requirement analysis for MindMate drew on three complementary sources. The first was the project brief supplied by the industry partner, which fixed the high-level objectives, the student-focused scope and the expectation of a privacy-respecting, locally runnable application. The second was a study of comparable digital mental-health tools and the academic literature surrounding them; systems such as Woebot, Wysa and Tess illustrate how conversational agents deliver structured, evidence-informed support to young adults (Fitzpatrick et al., 2017; Inkster et al., 2018; Fulmer et al., 2018), while broader reviews map the features that such tools typically provide and the safeguards they require (D'Alfonso, 2020; Torous et al., 2021). The third source was the body of clinical instruments and computational methods the system was to incorporate, namely the PHQ-9 and GAD-7 screening questionnaires (Kroenke et al., 2001; Spitzer et al., 2006), rule-based sentiment analysis (Hutto & Gilbert, 2014), and the machine-learning techniques provided by scikit-learn (Pedregosa et al., 2011).

From these sources a set of needs emerged. Students require a confidential, always-available space to express how they feel and to receive a supportive, non-judgemental response. They benefit from being able to record their mood over time and to see patterns in it. They need access to validated self-assessment of depression and anxiety symptoms, presented with appropriate care. They are better served when the support offered is tailored to their individual situation rather than generic, which motivates the machine-learning personalisation layer. They must be protected by an explicit safety mechanism for moments of crisis. Finally, the project itself requires the ability to evaluate whether the assistant makes a measurable difference, which introduces a requirement for structured data capture and an evaluation methodology.

These needs were then separated into functional requirements, which describe the concrete services the system delivers; non-functional requirements, which describe the quality attributes the system must possess; and user requirements, which describe the experience from the perspective of the student who uses it. The remaining sections of this chapter present each category in turn.

## 3.3 Functional Requirements

The functional requirements specify the observable behaviour of MindMate, the inputs it accepts, the processing it performs and the outputs it returns. They are enumerated in the table below and elaborated in the prose that follows. Each requirement carries a unique identifier (FR-1 to FR-12) so that it can be referenced from the design and testing chapters and traced to the use cases of Section 3.7.

| ID | Requirement | Description |
|----|-------------|-------------|
| FR-1 | Conversational support | The system shall provide an empathetic chat interface in which a student can type messages and receive supportive, context-aware replies, using an external language model when an API key is present and a fully offline rule-based engine otherwise. |
| FR-2 | Crisis-safety screening | The system shall screen every incoming user message for crisis risk before any reply is generated, classify the risk as none, elevated or crisis, and respond to a crisis with a validating safe message and helpline information rather than attempting counselling. |
| FR-3 | Mood logging | The system shall allow a student to record a daily entry comprising mood (1-5), energy (1-5), sleep hours and an optional free-text note, and to persist these entries for later review. |
| FR-4 | Sentiment analysis | The system shall compute a sentiment score for free-text notes and chat messages using VADER together with an emotion-keyword lexicon, and store the result alongside the text. |
| FR-5 | PHQ-9 depression screening | The system shall administer the nine-item PHQ-9 questionnaire on the standard 0-3 response scale, compute the total score and map it to the published severity band. |
| FR-6 | GAD-7 anxiety screening | The system shall administer the seven-item GAD-7 questionnaire on the standard 0-3 response scale, compute the total score and map it to the published severity band. |
| FR-7 | Self-harm safeguard | The system shall flag a non-zero response to PHQ-9 item 9 as a self-harm risk and surface a supportive safeguard message and helplines. |
| FR-8 | Student segmentation | The system shall assign a student to one of four support segments using a trained K-Means model over the six canonical wellbeing features. |
| FR-9 | Stress prediction | The system shall predict a student's stress level (Low, Moderate, High) and its class probabilities using a trained logistic-regression classifier over the six features. |
| FR-10 | Personalised recommendations | The system shall rank and present coping strategies using a transparent hybrid recommender that combines detected emotion, segment preference and stress level, and shall return an explanatory rationale for each recommendation. |
| FR-11 | Resources and psychoeducation | The system shall provide a resources page listing Indian and global helplines, psychoeducational content and the full coping-strategy library. |
| FR-12 | Data persistence | The system shall persist users, mood logs, chat messages, screening results and recommendation logs in a local SQLite database, with referential integrity between a user and their related records. |

The conversational requirement (FR-1) is the heart of the application. The chat must feel responsive and human, yet remain useful even with no internet connection or paid API access; the dual-engine design satisfies this by defaulting to a deterministic offline engine and upgrading to the external model only when a key is configured. Crucially, FR-2 takes precedence over FR-1 at runtime: no reply is composed until the crisis screen has classified the message, so a student in danger is never met with an ordinary chatbot answer. The screening is deliberately conservative, treating a false alarm as far less costly than a missed warning, in keeping with the duty-of-care principles emphasised in reviews of mental-health chatbots (Vaidyam et al., 2019).

The self-tracking and assessment requirements (FR-3 to FR-7) give the system its clinical grounding. Mood logging produces the longitudinal record that drives the trend charts and the engagement feature, while sentiment analysis enriches both notes and chat with an affective signal (Hutto & Gilbert, 2014). The PHQ-9 and GAD-7 instruments are implemented exactly to their published scoring rules and severity thresholds (Kroenke et al., 2001; Spitzer et al., 2006), and the item-9 safeguard (FR-7) mirrors the crisis layer's protective philosophy at the point of screening.

The machine-learning requirements (FR-8 to FR-10) deliver the personalisation that distinguishes MindMate from a generic chatbot. Segmentation and stress prediction operate over the six canonical features, average mood, average sleep hours, PHQ-9 score, GAD-7 score, average sentiment and engagement, and their outputs feed the recommender, which must remain explainable so that the support offered can always be justified to the student. Finally, the resources and persistence requirements (FR-11 and FR-12) ensure that help is one click away at all times and that every interaction is recorded in a structured form suitable for the project's evaluation.

## 3.4 Non-Functional Requirements

Non-functional requirements describe the quality attributes that determine how well the system performs its functions rather than which functions it performs. For a sensitive application of this kind they are as important as the functional behaviour itself.

**Performance.** The application must feel immediate during interactive use. Screening of a chat message, computation of sentiment, segment assignment and stress prediction must all complete within a fraction of a second on a standard laptop, since the underlying operations are lightweight rule evaluations and small scikit-learn model inferences over six features. The only operation with variable latency is a call to the external language model, which depends on network conditions; the offline engine therefore acts as a predictable fallback that keeps the interface responsive at all times.

**Usability.** The interface is intended for students who are not technical specialists and who may be using the tool while distressed. It must therefore be clean, calm and self-explanatory, with a small number of clearly labelled pages, plain-language prompts, and supportive rather than clinical tone. Streamlit's multipage navigation provides a consistent layout, and visualisations are used to make trends and machine-learning outputs intuitive rather than intimidating.

**Privacy and security.** Because the data is psychologically sensitive, privacy is a defining constraint. The system is designed to run locally, storing all personal data in a SQLite database on the user's own machine rather than transmitting it to a central server. The optional external language model is the only component that can send data off the device, and the application functions fully without it, so a privacy-conscious user can disable it entirely. No real identifying information is required to use the system, and all model development and evaluation in this project use clearly labelled synthetic data, avoiding any handling of real personal records.

**Reliability.** The system must behave predictably and must never fail in a way that leaves a distressed user without support. The offline engine guarantees that the chat always produces a reply; the crisis layer is rule-based and deterministic, so its protective behaviour does not depend on a probabilistic model; and a suite of automated tests verifies the critical paths. Reliability is reinforced by the safety design principle that errors should fail safe, for example, ambiguous messages err toward flagging rather than ignoring risk.

**Maintainability.** The codebase is organised into clear layers, presentation, a service layer, an assistant package and a machine-learning package over a database, so that each concern can be modified in isolation. Trained models are serialised with joblib and loaded at runtime, decoupling model training from application logic. A body of automated unit and integration tests (described in the testing methodology) protects against regressions and documents the intended behaviour, supporting the maintainable-design guidance of the software engineering literature (Sommerville, 2016).

**Portability.** The application depends only on Python 3.12 and a set of open-source, cross-platform libraries, with no platform-specific components. It therefore runs unchanged on Windows, macOS and Linux, and can be hosted on any environment that supports a Python web process. SQLite, being a single self-contained file, makes the data layer trivially portable as well.

## 3.5 User Requirements

User requirements describe what the system must offer from the standpoint of the people who use it, expressed in their own terms rather than in technical language. The primary user of MindMate is a higher-education student, and a representative persona helps to keep the analysis grounded.

**Persona.** Consider "Aditi", a twenty-one-year-old undergraduate in her second year. She is managing a demanding course alongside part-time work, sleeps irregularly during assessment periods, and notices that her mood and concentration suffer when deadlines cluster. She is reluctant to approach campus counselling because of stigma and waiting times, and she values privacy. She owns a laptop and is comfortable with everyday apps but is not technical. From Aditi's perspective the system must let her: talk to a supportive listener at any hour without being judged; quickly note how she is feeling and later see whether things are improving or worsening; check in a structured way whether her low mood or worry has reached a level worth taking seriously; receive concrete, manageable suggestions tailored to her situation rather than generic advice; and reach professional help immediately if she ever feels unsafe. The prevalence of mental-health difficulties among students reported in international surveys underlines that Aditi is far from unusual (Auerbach et al., 2018; Eisenberg et al., 2013), and the World Health Organization's call to widen access to support (World Health Organization, 2022) motivates a tool that meets students where they already are.

From this persona the following user requirements are drawn. The student requires a private and confidential environment that does not demand personal identification. They require an interface that is calming and easy to navigate even under stress. They require honesty about the tool's nature, it is a supportive companion and self-help aid, not a substitute for professional care or a diagnostic instrument, so that expectations are appropriately set. They require their data to remain under their control. And they require unambiguous, prominent routes to emergency help. These user requirements are deliberately satisfied by the functional and non-functional requirements above, ensuring traceability from human need to system behaviour.

## 3.6 Feasibility Study

A feasibility study examines whether the proposed system can realistically be built and operated given the available technology, budget and operating environment (Pressman & Maxim, 2014). For MindMate the study considers three dimensions.

### 3.6.1 Technical Feasibility

The project is technically feasible because every capability it requires is provided by mature, well-documented, open-source technology. The application is built in Python 3.12 with Streamlit for the user interface, scikit-learn for the machine-learning models, and VADER for sentiment analysis (Pedregosa et al., 2011; Hutto & Gilbert, 2014). These libraries are widely used, actively maintained and supported by extensive community knowledge, which substantially lowers the risk of encountering an insurmountable technical obstacle. The machine-learning components, K-Means clustering for segmentation (MacQueen, 1967; Lloyd, 1982) and multinomial logistic regression for stress classification, are standard algorithms with efficient implementations and modest computational demands, so they train and run comfortably on ordinary hardware. Data handling relies on pandas and numpy, persistence on the SQLite engine built into Python's standard library, and model serialisation on joblib, none of which introduces unusual complexity. The optional integration with an external language model is the only component that depends on a third-party service, and because a complete offline engine provides the same interface, the system carries no hard dependency on it. The presence of a passing automated test suite confirms that the intended behaviour is achievable in practice rather than only in principle. The project is therefore technically sound and well within the capability of the development environment.

### 3.6.2 Economic Feasibility

Economically the project is highly favourable, which is one of its deliberate design goals. The entire technology stack is free and open-source, so there are no licensing costs for the language, the web framework, the machine-learning libraries or the database. Development was carried out on hardware already available to the author, requiring no capital expenditure. The most significant potential running cost, fees for calls to a commercial language-model API, is avoided entirely by the offline rule-based engine, which allows the application to deliver its core conversational function at zero marginal cost. Where an external model is used, the lightweight default model keeps any expenditure minimal and entirely optional. Because the data is stored locally in a single SQLite file, there is no requirement for paid database hosting or cloud storage. The result is a system whose total cost of ownership is effectively nil for a student or institution, which is significant given that cost and access are recognised barriers to mental-health support (World Health Organization, 2022). The project is thus economically feasible with a wide margin.

### 3.6.3 Operational Feasibility

Operationally the system is straightforward to deploy and to use. Running it requires only a Python environment and the installation of the listed packages, after which a single command starts the Streamlit server and opens the application in a web browser. Because it runs locally and stores its own data, it needs no server administration, database management or network configuration for individual use; equally, because it is a standard Python web application, it can be hosted on any platform that supports such processes should wider access be desired. For the end user the operational burden is minimal: the interface is a familiar web page navigated through clearly labelled pages, demanding no training. The supportive, low-friction design and the privacy of local operation make adoption realistic for the student persona described earlier, who values confidentiality and ease of use. The system therefore fits comfortably into its intended operating context and is operationally feasible.

## 3.7 Use-Case Analysis

Use-case analysis models the system from the perspective of the goals that an external actor wishes to achieve through it. The single primary actor in MindMate is the Student, who interacts with all of the application's functions. A secondary, supporting actor is the external Language-Model Service, which the chat function may call upon when configured but on which the system does not depend. The use-case diagram below summarises the goals the student can pursue and their relationships to the system.

[[FIG: diagram_usecase.png | Figure 3.1: Use-case diagram of MindMate]]

The principal use cases are as follows.

1. **Chat with the assistant.** The student opens the chat page and exchanges messages with MindMate to obtain emotional support. Every message first passes through the crisis-screening use case; only then is a reply produced, either by the external service or by the offline engine. This is the most frequently exercised use case and the one that most directly delivers conversational support.

2. **Screen for crisis risk.** Invoked automatically for each chat message, this use case classifies risk as none, elevated or crisis. It is an included behaviour of the chat use case rather than something the student triggers explicitly, reflecting the safety-first runtime ordering. A crisis classification short-circuits the normal reply and presents a validating message together with helplines.

3. **Log mood.** The student records mood, energy, sleep hours and an optional note. The note is passed to the sentiment-analysis behaviour, and the entry is persisted for later review and for use as machine-learning features.

4. **Take a self-check.** The student completes the PHQ-9 or GAD-7 questionnaire, receives a total score and severity band, and, in the case of PHQ-9 item 9, may trigger the self-harm safeguard. The resulting scores are stored and feed into segmentation and stress prediction.

5. **View insights.** The student views their support segment, predicted stress level and its probabilities, the six feature values driving these outputs, and the rationale behind the recommendations. This use case makes the machine-learning reasoning transparent.

6. **Receive recommendations.** Drawing on detected emotion, segment and stress level, the system ranks coping strategies and presents them with an explanation. This behaviour is included by the chat and insights use cases.

7. **Access resources.** The student browses helplines, psychoeducation and the coping-strategy library. This use case is always available and is the destination to which the crisis and safeguard behaviours direct a user in need.

Together these use cases cover every functional requirement of Section 3.3, with FR-2 and FR-7 appearing as protective behaviours woven through the others rather than as standalone goals, an arrangement that captures the system's safety-first character.

## 3.8 Data-Flow Analysis

Data-flow analysis describes how information moves through the system, independent of the control structures that implement it. It complements the use-case view by focusing on the transformation of data rather than on user goals. The analysis is presented at two levels of abstraction.

The context, or Level 0, diagram treats the whole of MindMate as a single process and shows its boundary with the outside world. The sole external entity that initiates interaction is the Student, who supplies inputs, chat messages, mood entries and questionnaire responses, and receives outputs in the form of supportive replies, trends, scores, recommendations and resources. A second external entity, the optional Language-Model Service, exchanges prompt and completion data with the system only when configured. All persistent information is held in the local data store. The context diagram thus establishes the scope of the system and confirms that, apart from the optional model service, MindMate is a self-contained application that does not communicate with any external party.

[[FIG: diagram_dfd_level0.png | Figure 3.2: Data-flow diagram (Level 0 / context)]]

The Level 1 diagram decomposes the single process into the principal internal processes and shows the data stores that connect them. A chat message entering the system is first directed to the safety-screening process, which produces a risk level; depending on that level, the message is either answered by the crisis-response process or passed to the reply-generation process, which composes a supportive answer and consults the recommendation process. A mood entry flows into the mood-logging process, where the sentiment-analysis process derives an affective score before the entry is written to the mood-log store. Questionnaire responses flow into the screening process, which computes scores and severity bands and writes them to the screening-results store. The personalisation processes, segmentation and stress prediction, read the accumulated features from the data stores and produce a segment and a stress prediction, which the insights and recommendation processes consume. Each process writes its outputs back to the appropriate store, so that the database accumulates a complete, structured record of the interaction. The mapping between these processes and the database tables, users, mood logs, chat messages, screening results and recommendation logs, is direct, and the cascade relationships between a user and their records preserve integrity as data flows in over time.

[[FIG: diagram_dfd_level1.png | Figure 3.3: Data-flow diagram (Level 1)]]

Taken together, the two data-flow diagrams demonstrate that information passes through the system in a disciplined and traceable manner: every input is processed by an identifiable transformation, every output is derived from stored or computed data, and the safety screen sits squarely on the path of every chat message. This orderly flow underpins both the reliability of the application and the credibility of the evaluation that depends on the data it records.

## 3.9 System Requirements (Hardware/Software)

The final element of the analysis is a statement of the resources needed to develop, run and host MindMate. Because the system is deliberately lightweight, these requirements are modest.

**Hardware.** Development and execution require only a standard personal computer. A laptop with a dual-core processor, 8 GB of RAM and a few hundred megabytes of free disk space is more than sufficient, since the machine-learning models are small and operate over six features and a few hundred synthetic records. No graphics accelerator or specialised hardware is needed for either training or inference, and the SQLite database occupies negligible space. An internet connection is required only if the optional external language model is used; all other functions, including the offline chat engine, operate without one.

**Software.** The core software requirement is Python 3.12 together with the project's open-source dependencies: Streamlit for the web interface; scikit-learn for the K-Means and logistic-regression models, together with StandardScaler, Pipeline and PCA; vaderSentiment for sentiment analysis; pandas, numpy and scipy for data handling and statistics; plotly for in-application charts and matplotlib for report figures; joblib for model serialisation; python-dotenv for configuration; and SQLite, provided through Python's standard library, for persistence. The pytest framework supports the automated test suite and graphviz is used to render the structural diagrams. The application runs on any operating system that supports Python 3.12, Windows, macOS or Linux, and is accessed through a standard modern web browser. A code editor and the Git version-control system complete the development environment. This compact and entirely open-source software footprint is what makes the system portable, inexpensive and easy to reproduce, and it is consistent with the feasibility conclusions reached earlier in this chapter.
